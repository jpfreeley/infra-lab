# Epic E03: AWS Organization and Control Tower Governance Bootstrap

## Overview

Epic E03 focuses on establishing the foundational AWS Organization structure and governance controls using AWS Control Tower and Terraform. This includes bootstrapping the AWS Organization, importing and managing the Control Tower Landing Zone, and defining the core Organizational Units (OUs) to support a secure, multi-account SaaS environment.

---

### E03-S001: Create AWS Organization and enable all features

* Provisioned the AWS Organization resource with all features enabled (`feature_set = "ALL"`).

* Enabled service principals required by Control Tower and other AWS services.

* Configured the remote Terraform backend with S3 and DynamoDB locking for state management.

* Verified existing AWS Organization state and imported it into Terraform.

* Ensured Terraform state is securely stored and locked in S3 with cross-region replication and KMS encryption.

### E03-S009: Define and Attach SCPs

* Implemented baseline Service Control Policies (SCPs) to deny IAM user and long-lived credential management, deny disabling core security services (CloudTrail, Config, GuardDuty), and deny member accounts from leaving the AWS Organization.

* Applied SCP exemptions using `aws:PrincipalArn` with `ArnNotLike` condition to preserve operational access for:

  * `AWSControlTowerExecution`

  * `AWSCloudFormationStackSetExecutionRole`

  * `OrganizationAccountAccessRole`

  * `AWSReservedSSO_AWSAdministratorAccess_*` (IAM Identity Center administrator sessions)

* Attached the deny-leave-organization SCP at the Organization root.

* Attached IAM user restrictions to the `Workloads` and `Sandbox` OUs.

* Attached security service protection SCPs to the `Workloads`, `Infrastructure`, `Security`, and `Sandbox` OUs.

* Verified break-glass/admin path via `aws sts get-caller-identity --profile infra-lab` before applying to confirm SSO admin role exemption was correctly in place.

---

### E03-S010: Restrict Regions for Workloads

* Implemented a region restriction SCP to deny resource creation outside approved regions (`us-east-1`, `us-west-2`).

* Scoped the region restriction SCP to the `Workloads` OU to avoid impacting shared infrastructure and security administration.

* Used a `NotAction` allowlist for global and region-agnostic AWS services (IAM, STS, Route 53, CloudFront, Organizations, etc.) to avoid breaking foundational services.

* Reused the same SCP exemption model (`ArnNotLike` on `aws:PrincipalArn`) for break-glass and administrative access to avoid lockout during recovery or governance operations.

* Approved regions variable (`var.approved_regions`) defined for future extensibility.
