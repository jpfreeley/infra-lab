Background

Architecture Design and Objective

Enterprise-grade, secure, and scalable AWS platform architecture for autonomous AI execution with strict multi-tenant isolation and robust CI/CD.

The Infra Lab project is a comprehensive, enterprise-grade AWS platform architecture designed to build a secure, scalable, and multi-tenant infrastructure for an autonomous AI execution system. It features a multi-account AWS Organization with Control Tower, dual VPC architecture for strict environment isolation, Aurora Serverless with row-level security, internal PKI with mTLS for secure service communication, and a robust CI/CD pipeline with Terraform and CloudFormation. The project includes detailed agile backlogs with 236 stories across 12 epics covering everything from repo standards and Terraform foundations to observability, compliance, and cost management, aiming for a SOC2-ready, highly reliable, and scalable platform within a 90-day build timeline.

3 Dual VPC Architecture (per environment)

Each environment account (dev/staging/prod) contains two VPCs with a hard isolation boundary between them.

Control Plane VPC — 10.10.0.0/16 (The "Brain")
  • WAF → ALB (TLS termination)
  • ECS Fargate: API Service (Blue/Green via CodeDeploy)
  • Results Ingestion API (mTLS required)
  • RDS Proxy → Aurora Serverless v2 (Postgres, RLS enforced)
  • Secrets Manager, Step Functions, SQS Queues
  • VPC Endpoints (S3, SQS, ECR, CloudWatch, SecretsManager)
  • Private subnets only for DB and API
Execution Plane VPC — 10.20.0.0/16 (The "Red Zone")
  • ECS Fargate Worker Pools: nano / medium / xlarge (binpacked)
  • NAT Gateway (static egress IP — customer allowlist)
  • NACLs: block all RFC1918 egress (no internal scanning)
  • 3-layer safety guardrails: pre-dispatch scope, worker re-check, network deny
  • PrivateLink bridge to Results API (NO direct DB access)
  • Workers reach only: public internet targets + Results API

The Hard Boundary

Rule: Workers in the Execution VPC can NEVER connect directly to RDS. All findings flow via POST /findings to the Results Ingestion API over mTLS. Even a fully compromised worker cannot reach the database.

Connection Method: PrivateLink (recommended)

One-way traffic only. Execution VPC can call the Results API endpoint. Control VPC is not reachable from Execution VPC in any other way. No VPC peering broad enough to allow lateral movement.

RDS Proxy (AWS Managed — you do NOT build this)

RDS Proxy is a fully managed AWS service. You configure it (target DB, auth via Secrets Manager, subnets/SGs, pooling settings). AWS runs and scales the proxy fleet.
  • Connection pooling: Absorbs bursty Fargate connections — Postgres hates thousands of open connections
  • Failover handling: Holds app connections open during Aurora scaling/failover events
  • Security: DB credentials stored in Secrets Manager, rotated without app disruption
  • RLS compatibility: Use SET LOCAL app.tenant_id = ... per transaction (not session) to avoid tenant leakage across pooled connections

5 Row-Level Security (RLS)

PostgreSQL RLS makes the database itself enforce tenant isolation — not just application code. Even if a developer forgets WHERE tenant_id = ..., the DB blocks cross-tenant access.

Implementation Pattern

1) Table design

CREATE TABLE findings (
  tenant_id  uuid NOT NULL,
  job_id     uuid NOT NULL,
  run_id     uuid NOT NULL,
  finding_id uuid NOT NULL,
  severity   text NOT NULL,
  evidence_s3_path text,
  PRIMARY KEY (tenant_id, finding_id)
);

-- Composite indexes start with tenant_id
CREATE INDEX findings_tenant_job_idx ON findings (tenant_id, job_id);
CREATE INDEX findings_tenant_run_idx ON findings (tenant_id, run_id);

2) Separate owner vs runtime role

CREATE ROLE Platform_owner NOLOGIN;   -- migrations only
CREATE ROLE Platform_app LOGIN PASSWORD '...';  -- runtime API

GRANT SELECT, INSERT, UPDATE, DELETE ON findings TO Platform_app;

3) Enable + FORCE RLS

ALTER TABLE findings ENABLE ROW LEVEL SECURITY;
ALTER TABLE findings FORCE ROW LEVEL SECURITY;

4) Define policy

CREATE POLICY findings_tenant_isolation
ON findings FOR ALL
USING (
  tenant_id = current_setting('app.tenant_id', true)::uuid
)
WITH CHECK (
  tenant_id = current_setting('app.tenant_id', true)::uuid
);

5) Set tenant context per request (transaction-scoped)

BEGIN;
SET LOCAL app.tenant_id = '7c2c0f0f-4f6f-4b6f-9a22-6d0d7a9f0b2a';
SELECT * FROM findings WHERE job_id = '...';
COMMIT;
Why SET LOCAL? It resets automatically at end of transaction — critical for connection pooling (RDS Proxy / PgBouncer) to prevent tenant context leaking to the next request.

Common Pitfalls

PitfallFix
App connects as table ownerSeparate owner vs runtime role
Role has BYPASSRLSNever grant BYPASSRLS to app role
Forgot FORCE ROW LEVEL SECURITYAlways FORCE, not just ENABLE
SET (session) instead of SET LOCAL (transaction)Always use SET LOCAL inside transaction
SECURITY DEFINER functions bypass RLSTreat as privileged code, minimize usage

6 Internal PKI + mTLS

Internal PKI is a system for creating, managing, and revoking digital certificates trusted only within your own network. mTLS (Mutual TLS) means both sides of a connection present a certificate — not just the server.

Why Platform needs it

The Problem: If a worker in the Execution VPC needs to POST findings to the Results API in the Control VPC, how does the API know it is really your worker and not an attacker who hopped onto your network?

The Solution: The API requires a client certificate. The worker presents one. Both certificates are issued by your Internal Root CA.

AWS Private CA (the managed approach)

  • Root CA: Created in the Security or Infrastructure account. KMS/HSM-backed (FIPS 140-2 Level 3). Keys never leave the hardware.
  • Issuing CA: Middle-man CA that signs daily worker certificates. Can be revoked without destroying Root.
  • RAM sharing: Private CA shared to dev/staging/prod accounts via AWS Resource Access Manager.
  • ACM integration: Fargate tasks request private certs automatically at startup via ACM.
  • Short-lived certs: Rotate automatically — a stolen container image alone is useless without a valid cert.

What an HSM is (and why it matters)

An HSM (Hardware Security Module) is a physical vault for cryptographic keys. The private key never leaves the hardware — even AWS admins cannot export it. AWS KMS uses HSMs behind the scenes. For Platform's Root CA, the key is hardware-bound and non-exportable.

mTLS Flow: Worker → Results API

  1. Worker starts in Execution VPC
  2. Worker requests a short-lived cert from ACM (backed by Private CA)
  3. Worker calls POST /findings presenting its client cert
  4. Results API validates the cert against the Private CA trust chain
  5. Only if valid: findings are written to DB
  6. Cert expires in hours — rotation is automatic

11 Interview Narrative

The Vision

"When I look at Platform, I don't see a collection of security tools — I see a high-scale distributed system that happens to perform offensive security. To move from $8M seed to enterprise traction, we have to solve for Trust and Scale simultaneously. My 90-day mission is to transform the current Docker-Compose on EC2 setup into a zero-trust, multi-tenant architecture that can handle 1,000 concurrent deep-Autonomous Executions without breaking a sweat or risking customer data."

The Foundation (Weeks 1–3)

"The first thing I'm doing is establishing a hard isolation boundary. I'm implementing AWS Control Tower to move us into a true multi-account structure: Dev, Staging, and Prod, plus dedicated Audit and Log-Archive accounts. Inside each environment, I'm deploying a Dual-VPC architecture — the Control Plane VPC (the Brain) and a completely separate Execution Plane VPC (the Red Zone). By using NACLs to block all internal RFC1918 traffic and forcing workers to communicate back only via PrivateLink and a specific Results Ingestion API, we ensure that even if a worker is compromised by a malicious target, the Brain remains unreachable."

The Engine (Weeks 4–7)

"To handle the unpredictable load of autonomous Execution, I'm moving us to ECS Fargate. Synchronous dashboard/API traffic sits behind an ALB with Blue/Green deployments. The actual Execution work is orchestrated by Step Functions and SQS across three worker tiers — Nano, Medium, and X-Large — to binpack compute costs. This keeps COGS low while giving us infinite headroom to scale."

The Security (Weeks 8–10)

"Because we are a security company, good enough encryption is not enough. I'm implementing Internal PKI using AWS Private CA. Every worker will be issued a short-lived identity certificate at runtime. The Results API requires mTLS — only a cryptographically verified Platform worker can write findings to our database. Every finding is stamped with tenant_id, job_id, and module_version — a perfect audit trail."

The Maturity (Weeks 11–12)

"I'm closing the loop with OpenTelemetry — full distributed tracing from the moment a user clicks Scan to the moment a worker sends a packet. The real founding engineer feature is our Versioned Rollout mechanism: SSM Parameter Store gates worker versions globally. We can flip the entire fleet or roll back in seconds without a redeploy, using a Canary Tenant model."

Pressure Test Answers

Q: Why Terraform AND CloudFormation?

Terraform for stateful infra (VPCs, RDS, SQS) — superior state management and multi-account provider ecosystem. CloudFormation for app-native pieces like ECS Blue/Green CodeDeploy — AWS-native tools handle wait-for-success and auto-rollback more reliably than a third-party state engine.

Q: Why not Kubernetes (EKS)?

At our current stage, EKS is an operational tax we don't need to pay. ECS Fargate gives us the same container isolation and scaling benefits with 90% less configuration overhead. Our innovation tokens should be spent on The Engine's AI Execution capabilities, not managing a K8s control plane. We've built the VPCs to be EKS-ready if we ever need it.

Q: How do you handle the Noisy Neighbor problem?

Three levels: Compute (Fargate tasks are kernel-isolated), Queueing (per-tenant SQS backlogs if one customer floods us), Database (Postgres RLS ensures a code bug can't leak data from Tenant A to Tenant B).

Mic Drop Closing Statement

"By Day 90, Platform won't just have infrastructure. We will have a Security Fortress — SOC2-ready on day one, clear margin visibility per customer to the CEO, and the engineering team with the autonomy to ship new AI Execution modules with total confidence that the underlying plumbing is indestructible."

13 Critical Questions to Ask the CTO

A. Long-Running Execution State

"As The Engine moves to complex multi-step autonomous Taskation, some jobs might take 12+ hours. How are we currently persisting the state of an ongoing attack? If a Fargate worker is preempted, do we restart the entire job or do we have a checkpointing mechanism?"

B. Data Residency vs. Global Scale

"As we sign European or Financial Services customers, will we be required to keep findings and evidence within specific geographic regions? Does our current Control Plane support a multi-region Cell-based architecture, or are we hard-coded to us-east-1?"

C. The AI Feedback Loop

"Does the infrastructure need to support a Training Loop where successful Tasks are fed back into a model? Do we need a high-throughput pipeline (Kinesis/Kafka) to move data from the Execution Plane into a Data Lake for ML engineers?"

D. Authenticated Scanning + Secrets Handling

"Are we planning to support Authenticated Scans where customers give us SSH keys or web session tokens? If so, how are we injecting those into ephemeral Fargate workers without them ever touching a disk or being logged? (Nitro Enclaves?)"

  • Ensure cost allocation tags are enforced in CI/IaC; block untagged resources in prod where possible.
  • Build tenant-level and job-level cost reporting (even if approximate) and publish a monthly margin report.
  • Set cost guardrails: max concurrency per tenant, max spend per day, and alerts on anomalous cost spikes.

Success evidence: a dashboard/report that answers How much did tenant X cost? and What did job Y cost?.

Compliance Readiness (Target: Level 4→5 by Day 90)
  • Automate evidence collection (where practical) for access, logging, and change management.
  • Document data lifecycle: evidence storage (S3), retention, encryption, deletion, and audit trail.

Success evidence: repeatable evidence workflow + documented policies + audit dry-run notes.
